Service Worker — Caching Strategy
App Shell — Cache First
What: HTML, CSS, JS, fonts, icons
Strategy: Precache on install, update on new deploy
Result: App loads instantly, even offline
Thumbnails — Stale While Revalidate
What: /api/thumbnail/* responses
Strategy: Serve from cache, fetch fresh in background
Result: Instant thumbnail display, offline browse of viewed albums
API Data — Network First
What: Album lists, media metadata, comments
Strategy: Try network, fallback to cache
Result: Fresh data when online, cached data when offline
Original Media — Network Only
What: Full-res photos, videos (signed URLs)
Strategy: No caching (large files, signed URLs expire)
Result: Always fresh, no storage bloat
Cache Storage Budget

Offline UX — What Users See
✅ Works Offline
• App opens & navigates
• Browse previously viewed albums
• View cached thumbnails
• See cached media metadata
⚠️ Degraded Offline
• Albums not yet viewed → empty
• Search → "Offline" message
• Comments → cached snapshot
• Full-res photo → "Unavailable offline"
❌ Requires Internet
• Upload photos/videos
• Post comments
• Manage members
• Login/register
• Video streaming

Implementation — next-pwa (Serwist)

Library
@serwist/next (successor of next-pwa, actively maintained)
SW Registration
Auto-registered via @serwist/next plugin in next.config.ts
Precache
App shell auto-precached from Next.js build output
Runtime Cache
Custom routes: thumbnail (StaleWhileRevalidate), API (NetworkFirst)
Cache Eviction
Thumbnails: LRU, max 1500 entries or 50MB, expire after 30 days
Update Flow
New SW detected → toast "Update available" → user clicks → reload
Offline Indicator
Top banner "You're offline — showing cached content" when navigator.onLine = false
